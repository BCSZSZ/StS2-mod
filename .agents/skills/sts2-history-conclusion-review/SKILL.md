# StS2 Conclusion Review

Use this skill for the explicit reasoning layer. This is where Codex or a human reviewer turns generated evidence candidates into reviewed conclusions.

## Inputs

- Generated conclusion candidates, or current substitutes: `28_card_evidence.csv`, `29_rules_library.csv`, `30_rule_validation.csv`
- Guardrail outputs from data credibility and stratification checks
- Source CSV rows referenced by each claim

## Review States

- `accepted`: enough evidence and stable enough for the final report
- `downgraded`: useful but weaker than the generated strength implies
- `sample_hint`: interesting but too small or unstable
- `rejected`: contradicted, misleading, duplicate, or not actionable
- `needs_more_data`: missing source rows or missing split checks

## Task Flow

1. Review one claim at a time.
2. Read its evidence and counter-evidence, not just the generated summary sentence.
3. Check sample count, version stability, act/floor stability, and directness of metric.
4. Decide review state and final strength.
5. Write a concise reason that names the evidence, the limitation, and the strategic reading.
6. Keep reviewed output structured so the final report can be generated from accepted rows.

## Code Boundary

If reviewed conclusions need a durable artifact, add a generated or curated table path such as `reports/strategy/reviewed_conclusions.csv` and a markdown companion. Keep raw metrics generated by Python; keep review decisions explicit and traceable.

## Reasoning Rule

This is one of the few steps where reasoning is required. Do not pretend code alone can decide causality, strategic meaning, or final wording.
