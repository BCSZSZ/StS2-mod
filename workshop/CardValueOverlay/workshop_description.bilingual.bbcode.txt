[h1]Card Value Overlay[/h1][i]English / 简体中文[/i][h2]Overview[/h2]Card Value Overlay uses mathematical modeling and simulation to display card play value, deck-level value changes, and historical choice data in Slay the Spire 2. It currently focuses on Regent and Colorless cards. Complex effects such as potion generation, random generation, selection, and transformation may not yet be quantified reliably.[h2]Requirements and usage[/h2][list][*]Requires BaseLib.[*]Displays card values above cards.[*]calc and dEV may briefly show ... while background simulation runs. n/a means the situation could not be evaluated reliably.[*]Real-time simulation parameters can be adjusted in the BaseLib mod configuration.[/list]This is a client-side display mod: it does not alter multiplayer game state, and teammates do not need to install it.[h2]Display guide[/h2][list][*][b]short / mid / long[/b]: 4 / 8 / 14-turn horizons.[*][b]est[/b]: static card play value from the built-in value model and calibration.[*][b]calc[/b]: real-time card play value in the current deck, on the same scale as est.[*][b]dEV[/b]: estimated deck-level value change after adding the card. In deck or upgrade views it compares keeping the copy with removing it.[*][b]after[/b]: estimated total deck value after the displayed comparison.[/list]calc and dEV are local and short-sighted: they evaluate only the current deck and selected horizon, not the entire future run.[h2]Historical statistics[/h2]Based on 956 publicly available Regent A10 solo standard wins on v0.107.0/v0.107.1.[list][*][b]deck[/b]: percentage of winning final decks containing the card, ignoring upgrade form.[*][b]p+0 / p+1[/b]: reward-only pick probability for the unupgraded / upgraded form. Enchantments are not separated.[*][b]b+0 / b+1[/b]: merchant-only buy probability for the unupgraded / upgraded form. Enchantments are not separated.[*][b]copy[/b]: average copies among final decks already containing the card; +0 and +1 are combined. Color thresholds exclude starter cards and require appearance in at least 30 final decks.[/list]p and b use separate offer denominators and separate Q25/Q75 thresholds. For est, calc, and dEV, green / red means positive / negative. For deck, pick, buy, and copy, filtered empirical Q25/Q75 thresholds are used: green is the top 25%, red is the bottom 25%. Ancient pick is the historical probability that an offered option was chosen.[h2]Future plans[/h2][list][*]Support relics in value calculations.[*]Add value calculations for other characters.[*]Further improve simulator logic.[/list][h2]Simulator details[/h2]The simulator's detailed calculation logic will be documented separately. Source code and data-processing scripts may also be inspected freely.[h2]Credits and feedback[/h2]The concept was inspired by Solisora's card-value framework, while the modeling, implementation, and calculations are my own. Errors reflect limitations in my work, not a failure of Solisora's method. Feedback is welcome.[hr][/hr][h1]Card Value Overlay 中文说明[/h1][h2]简介[/h2]本 Mod 通过数学建模和模拟计算，在 Slay the Spire 2 中显示卡牌打出价值、卡组整体价值变化和历史选择数据。目前主要支持储君与无色卡；炼药、随机生成、选择和变形等复杂效果仍可能无法可靠量化。[h2]前置与使用[/h2][list][*]前置需求：BaseLib。[*]在卡牌上方显示价值数据。[*]calc 与 dEV 在后台计算时可能短暂显示 ...；n/a 表示当前场景无法可靠计算。[*]可在 BaseLib 的 Mod 配置中调整实时模拟参数。[/list]本 Mod 仅在本地显示信息，不修改多人游戏状态，联机队友不需要安装。[h2]显示内容[/h2][list][*][b]short / mid / long[/b]：4 / 8 / 14 回合视角。[*][b]est[/b]：由内置静态价值模型和校准参数计算的卡牌打出价值。[*][b]calc[/b]：当前卡组语境下实时计算的卡牌打出价值，与 est 同量纲。[*][b]dEV[/b]：加入该卡后的卡组整体价值变化；在卡组或升级界面表示保留该牌相对于移除该牌的变化。[*][b]after[/b]：当前比较下的卡组总价值估计。[/list]calc 和 dEV 只评估当前卡组与所选视角，不代表整局游戏的长期发展。[h2]历史数据[/h2]数据来自 956 场 v0.107.0/v0.107.1、A10、单人、标准模式的储君胜利局。[list][*][b]deck[/b]：该卡出现在最终通关卡组中的比例，不区分升级形态。[*][b]p+0 / p+1[/b]：该卡以未升级 / 升级形态出现在卡牌奖励时被选择的概率，不区分附魔。[*][b]b+0 / b+1[/b]：该卡以未升级 / 升级形态出现在商店时被购买的概率，不区分附魔。[*][b]copy[/b]：在最终卡组已包含该卡的情况下平均有几张；+0 与 +1 合并。颜色分位排除初始牌，并要求至少出现在 30 个最终卡组中。[/list]p 与 b 使用各自独立的展示分母和 Q25/Q75 颜色阈值。est、calc 和 dEV 的绿色 / 红色表示正数 / 负数。deck、pick、buy 和 copy 使用过滤后的经验 Q25/Q75：绿色为前 25%，红色为后 25%。先古选项的 pick 表示该选项出现时被选择的历史概率。[h2]未来计划[/h2][list][*]数值计算支持遗物。[*]追加其他职业的数值计算。[*]进一步优化模拟器逻辑。[/list][h2]模拟器详细逻辑[/h2]模拟器的详细计算逻辑会另行说明。源码和数据处理脚本也欢迎自由查看。[h2]致谢与反馈[/h2]本 Mod 的思路受到 Solisora 价值化卡牌体系启发，具体建模、实现与数值计算由我独立完成。若数值或结论有误，应归因于我的建模和实现不足，并不代表 Solisora 的方法或价值体系不成立。欢迎提出建议。
