# StS2 Evidence Chain

Use this skill for stage 8: converting tables into auditable conclusion candidates.

## Inputs

- V2: `28_card_evidence.csv`, `29_rules_library.csv`, `30_rule_validation.csv`
- Supporting V1/V2 rows from any table named in a candidate's `source_table` or `evidence_refs`

## Desired Candidate Schema

When implementing or extending the conclusion layer, prefer a generated table like:

`candidate_id`, `stage`, `subject_type`, `subject_id`, `claim`, `evidence_refs`, `counter_evidence_refs`, `metric_snapshot`, `strength`, `needs_reasoning`, `review_status`, `source_tables`

## Task Flow

1. Start from generated V2 rules and card evidence.
2. Attach precise evidence references to source tables and rows.
3. Attach counter-evidence when version, act, floor, or pairwise checks disagree.
4. Assign initial strength mechanically from sample gates and stability checks.
5. Mark `needs_reasoning=true` for causality, context-dependent card evaluation, matchup interpretation, and final strategic wording.
6. Keep all candidate rows generated by code; review decisions belong in the conclusion-review step.

## Code Boundary

If conclusion candidates do not exist yet, implement them in `src/history_analysis/strategy.py`, validate schema in tests, and regenerate V2. Do not bury conclusion candidates only in markdown prose.

## Output Rule

This step produces auditable candidates, not final strategy truth. A claim enters the final report only after conclusion review accepts it.
